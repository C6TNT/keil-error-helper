# AI接入设计

这份文档的目标不是直接接某一家模型接口，而是先把 `keil-error-helper` 在接入 AI 前最关键的输入、输出和调用边界定义清楚。

当前产品已经具备：

- 第一条错误提取
- 规则分类
- 模板层定位
- 场景选择
- 求助文本生成
- 新手卡片展示

所以后续接入 AI 时，不建议把原始编译输出直接整段扔给模型自由发挥，而应该采用：

**规则先定位，AI 再增强解释。**

---

## 1. AI 在产品中的定位

AI 不负责替代规则引擎，也不负责第一时间决定“这是什么报错”。

AI 更适合负责这些事：

1. 把当前这条错误解释得更容易懂
2. 结合“页面 / 按键 / 参数 / 频率 / 超声波”等场景，给更贴近比赛的排查建议
3. 帮用户缩小排查范围
4. 生成更自然的求助文本
5. 在用户补充代码片段后，给出更具体的同步修改建议

---

## 2. 推荐的 AI 调用时机

建议分层调用，不要默认每次都直接请求 AI。

### 默认本地分析

用户点击“开始分析”后，先始终走本地规则引擎：

- 提取第一条错误
- 匹配错误类型
- 生成卡片
- 生成求助文本

### AI 深入分析

只有用户主动点击“AI 深入分析”时，才调用模型。

这样做的好处：

- 没网时基础功能仍然可用
- 成本可控
- 响应更快
- AI 输出更有边界

---

## 3. AI 输入模板

建议统一使用结构化输入，避免不同页面、不同调用点各自乱拼 prompt。

### 最小输入字段

```json
{
  "product_name": "keil-error-helper",
  "version": "V0.2",
  "error_found": "yes",
  "error_raw": "App\\app.c(128): error C208: '_SEG_SetCode': too many actual parameters",
  "error_code": "C208",
  "error_file": "App\\app.c",
  "error_line": "128",
  "rule_id": "too_many_params",
  "rule_title": "函数参数个数不匹配",
  "template_area": "业务层（App）",
  "scene": "display",
  "scene_label": "显示",
  "priority_level": "高概率问题",
  "priority_text": "这类错误通常定位很集中，修掉它之后，后面一串连带报错往往会明显减少。",
  "pitfall_title": "你可能只改了 .c 里的调用，没有同步 .h 里的声明。",
  "report": "完整本地分析结果",
  "feedback_text": "当前生成的求助文本",
  "raw_build_output": "完整 Keil 编译输出"
}
```

### 可选增强字段

如果后面支持“补充代码片段”，可选加入：

```json
{
  "code_context": {
    "file": "App/app.c",
    "snippet": "相关代码片段",
    "user_note": "我刚把第十届示例迁移进来"
  }
}
```

## 4. 推荐的 AI 系统提示词结构

建议把 AI 的职责限制得很清楚，不要让它自由发挥成泛泛而谈的代码助手。

### 系统提示词建议

```text
你是一个面向蓝桥杯单片机新手的 Keil/C51 报错诊断助手。
你已经拿到了本地规则引擎整理好的错误信息。
你的任务不是重复解释所有内容，而是：
1. 用更容易懂的话说明这条错误大概是什么意思。
2. 结合当前场景（页面、按键、参数、温度、频率、超声波、显示）给出更贴近比赛代码的排查建议。
3. 明确告诉用户先看哪一段、先做哪一步。
4. 不要泛泛而谈，不要给太长背景介绍。
5. 优先支持零基础新手。
6. 如果信息不够，不要编造，明确指出还需要什么信息。
```

### 用户消息建议

直接发送结构化 JSON，或者“短说明 + JSON”。

例如：

```text
请根据下面的结构化错误信息，给出更适合蓝桥杯单片机新手的解释和排查建议：

{...json...}
```

---

## 5. AI 输出建议格式

建议 AI 输出固定成 3 段，不要聊天化。

### 建议输出模板

1. 这条错误更像什么问题
2. 你应该先看哪几处
3. 如果你刚在改某个模块，最可能漏改哪里

示例：

```text
这条错误更像是函数参数没有同步。

你应该先看：
1. 当前调用位置
2. 对应 .h 里的函数声明
3. 对应 .c 里的函数定义

如果你刚在改显示模块，这类错误最常见的是：
- 你把显示函数改成了新参数
- 但旧的页面显示函数还在按旧参数调用
```

## 6. 产品路线建议

### V0.3

- 在界面里新增 “AI 深入分析” 按钮
- 先不接代码片段，只传当前结构化报错信息
- 返回 3 段简明建议

### V0.4

- 支持用户附加“我刚改的代码片段”
- AI 可以更具体地判断是声明、定义还是调用没同步

### V0.5

- 支持把 AI 结果和本地结果融合成更统一的结果卡片
- 支持生成更像“学长回复”的修复建议

### V1.0

- 形成“本地规则分析 + AI 深入建议 + 求助文本生成”的完整闭环

---

## 7. 当前最推荐的接入方式

如果下一步真的开始接模型，建议优先：

1. 保留现有本地规则分析不动
2. 新增一个“AI 深入分析”按钮
3. 把当前分析结果整理成固定 JSON 输入模板
4. 让 AI 只做增强解释，不接管基础诊断

这样最稳，也最符合这个产品的定位。
